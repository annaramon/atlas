Confirmed decisions

  • ✓
    Ollama only — no Anthropic API
    Fully local stack. Health data stays on machine. Switch to Claude later by changing 3 files (requirements.txt, config.py, agent.py).
  • ✓
    No computed/derived metrics in MVP
    Agent queries raw DB data directly. processing/ folder is intentionally absent.
  • ✓
    Activity aggregation at query time
    Store raw 30-min slots. Daily totals computed by the agent tool on demand.
  • ✓
    Skip GPX files silently
    FIT files are the source of truth. Pipeline will filter .gpx by extension.
  • ✓
    Skip gears.json and reactions.json
    reactions.json is empty. gears.json is equipment tracking — out of scope for MVP.
  • ✓
    No test abstraction layer for LLM
    Mock the Ollama client directly in tests. Tools are tested against a real test DB without touching the LLM at all.

Risks & things to fix

With no derived metrics in MVP, pandas and numpy are only useful for query-time aggregation in tools. This is achievable with plain SQLAlchemy + Python.
Item Severity Detail Action
Model version mismatch high .env.example says qwen2.5:32b, CLAUDE.md setup command says ollama pull qwen2.5:72b. These are different models. Decide which model to standardise on and fix both files before first run.
alembic.ini hardcodes DB credentials medium sqlalchemy.url in alembic.ini is hardcoded to atlas:atlas@localhost. This diverges from .env and breaks if credentials ever change. In alembic/env.py override sqlalchemy.url from config.DATABASE_URL at runtime.
alembic/env.py is empty medium Alembic won't discover models or connect to the DB until env.py and script.py.mako are written. Running alembic upgrade head right now fails silently. Write these as part of the core setup step.
HR fraction normalization medium sleep.hrAvg = 1.07 — stored as a fraction of hrMax, not bpm. Multiplying by 198 gives ≈212 bpm which is above hrMax (198). The normalization formula needs validation against actual known sleep HR values. Sample 2–3 nights and verify the formula before ingesting all data.
No virtual environment in repo medium No .venv/, no Makefile, no setup instructions beyond CLAUDE.md. A fresh clone has no clear pip install step documented. Add a make install target or a one-liner in CLAUDE.md: python -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt.
Ollama tool calling reliability medium Local models are less reliable at multi-turn tool use than Claude. qwen2.5 may produce malformed tool call arguments or fail to call the right tool for ambiguous queries. Design tool schemas to be as explicit as possible. Add input validation in each tool function. Log all tool calls during development.
No idempotency strategy yet medium Re-running the ingestion pipeline on the same files will produce duplicate rows. Strategy (unique constraint + upsert) is documented but not yet implemented. Enforce UNIQUE on workout.source_file, sleep_session.sleep_id, and activity_slot.timestamp at migration time.
pandas/numpy in requirements but no usage planned low Keep them — they're lightweight and useful for future aggregations — but don't depend on them for MVP logic.
No health check on POST /ask low If Ollama is not running, the endpoint will fail with an unhelpful connection error. Fine for MVP single-user local use but annoying in practice. Add a startup check or a GET /health endpoint that pings both Postgres and Ollama.
data/raw/ folder exists but is unused info Created as a future drop zone but the current export lives at repo root. Could cause confusion about where to put new exports. Either remove it or add a note in CLAUDE.md clarifying its future purpose.
sleep_stages/ data only starts 2025-09 info sleep/ summaries are also only from 2025-09. activity/ starts 2024-12. Temporal gaps are expected but the agent needs to handle "no data for that period" gracefully. Each tool should return a clear empty state rather than raising an error when the date range has no data.

Recommended implementation order

#StepFilesWhy first
1 Core config + DB session core/config.py, core/database.py Everything depends on this. Write once, never touch again.
2 Models + Alembic migration models/*.py, alembic/env.py, alembic/versions/ Tables must exist before ingestion or tests can run.
3 Ingestion pipeline ingestion/fit_parser.py, ndjson_parser.py, pipeline.py Without data in the DB, the agent has nothing to query.
4 Agent tools agent/tools.py Pure SQLAlchemy functions — testable in isolation before adding Ollama.
5 Agent loop agent/agent.py Wire Ollama to the tools. First end-to-end query.
6 API endpoint main.py, api/ask.py Thin wrapper. Last because everything it calls is already tested.
