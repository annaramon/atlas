Atlas

Personal physiological intelligence agent for an elite multi-sport athlete.
A single POST /ask takes a natural language question and returns a data-grounded answer drawn from real Suunto wearable data.

What is Atlas?

Atlas is a local AI agent that answers questions about training load, sleep quality, recovery state, and daily activity — using data exported from a Suunto wearable device. Everything runs on your machine: no cloud API, no third-party data transfer.

A typical interaction looks like this:

# You ask:
"How was my sleep and recovery over the past two weeks, and how did my hard workouts affect them?"

# Atlas fetches sleep rows, recovery rows, and workout rows from the DB,
# passes them to the model, and returns a specific, data-grounded answer:
"Your average sleep quality was 0.78 over the past 14 days.
 The three nights after Zone 4/5 sessions (Mar 3, Mar 7, Mar 11)
 showed 12% lower HRV and recovery balance dropped to 0.51–0.58.
 Deep sleep remained consistent at ~58 min per night."

The key design decisions are:

  • Local LLM. Ollama runs qwen2.5:32b on your hardware. Your physiological data never leaves the machine.
  • No pre-computation. The agent queries raw stored data and aggregates at query time. This keeps ingestion simple and avoids derived metrics that could be wrong.
  • Tool-calling, not RAG. The model doesn't search a vector store — it calls typed functions that return structured JSON from the database. This is more reliable for numerical data.

These layers only communicate in one direction. The API never touches the ingestion code. The agent never knows about the file formats. Each layer can be understood, tested, and replaced independently.

Data layer

SQLAlchemy ORM models define the schema. Alembic manages migrations. The ingestion pipeline is the only writer — everything else is read-only.

Agent layer

The LLM drives a tool-calling loop. It decides which tools to call and what arguments to pass. Tools are typed Python functions that return plain dicts.

API layer

A single FastAPI endpoint accepts a question string, hands it to the agent, and returns the answer. No auth, no sessions — single-user MVP.

Infrastructure

Postgres runs in Docker. Ollama runs natively. The app reads config from .env via Pydantic Settings.

ORM Models app/models/

SQLAlchemy models define the shape of the database. They are the shared language between the ingestion pipeline (which writes) and the agent tools (which read). Changing a model without a migration would silently break both.

Why SQLAlchemy ORM?

Raw SQL strings in query functions would work, but they are invisible to the type checker, fragile to rename, and hard to test without a real database. ORM queries are Python — they compose, they're readable, and SQLAlchemy translates them correctly for both Postgres (production) and SQLite (tests).

The four tables

ModelTableKey columnsUnique constraint
Workout workouts sport_type, total_duration, hr_avg/max/min, hr_zone_1..5 source_file — prevents re-ingesting the same FIT file
Sleep sleep deep/light/rem_duration, hr_avg, hrv, spo2_avg, quality_score recorded_at — one session per timestamp
SleepStage sleep_stages sleep_id (FK), stage (deep/light/rem/awake), duration (sleep_id, recorded_at)
Activity activity step_count, energy_consumption recorded_at — 30-min slots, one per slot
Recovery recovery balance (0–1), stress_state recorded_at
Why store activity as 30-min slots? The Suunto export writes one NDJSON row per 30-minute window. Storing them raw means the ingestion is a simple, lossless pass-through. The agent tool aggregates them into daily totals at query time using func.sum() + GROUP BY day. If the aggregation logic ever needs to change, you change one line in tools.py, not re-ingest 34,000 rows.

The SleepStage relationship

Sleep sessions and granular stage data live in separate files in the Suunto export (sleep/ and sleep_stages/). The SleepStage model has a foreign key to Sleep.id and a SQLAlchemy relationship for ORM navigation. Linking happens during ingestion by matching timestamps: any stage row whose recorded_at falls within a session's [bedtime_start, bedtime_end) window belongs to that session.

Ingestion Pipeline app/ingestion/

The pipeline is a one-time (or re-runnable) script that reads the raw Suunto export directory and populates the database. It is the only place that knows about file formats.

# Run it:
python -m app.ingestion.pipeline suunto-data-03042026

Structure

Three files, one responsibility each:

  • fit_parser.py — parses a single FIT binary file into a dict matching the Workout model
  • ndjson_parser.py — parses individual lines from sleep, activity, and recovery NDJSON files
  • pipeline.py — orchestrates: discovers files, calls parsers, inserts rows, commits

Why is the pipeline separated from the parsers?

A parser function takes raw data and returns a dict. It has no side effects and no database dependency. This makes it trivially unit testable — you pass in a sample dict, you assert the output, no DB setup required. The pipeline is the only place that opens DB sessions and calls INSERT.

Idempotency

Every insert uses ON CONFLICT DO NOTHING via SQLAlchemy's PostgreSQL insert() dialect. Running the pipeline twice on the same data produces the same result — no duplicates, no errors. This matters because re-running after adding new months of data should be safe.

# Workout insert — source_file is the unique key
db.execute(
    insert(Workout).values(**row).on_conflict_do_nothing(index_elements=['source_file'])
)

FIT parser (fit_parser.py)

FIT is a binary format from Garmin (used by Suunto). The fitparse library decodes it. The parser extracts the session message — the single record that contains aggregate workout metrics. Individual data points (GPS track, per-second HR) are not stored; only the summary.

Sport type and timestamp come from the filename, not the FIT session message. This is intentional: some sport types like gym report as training in the FIT metadata, which loses useful information. The filename (2024-12-27_10.26.13-cross_country_skiing.fit) is authoritative.

GPX files are silently skipped. Suunto exports both a FIT and a GPX file per workout. GPX contains only GPS track data, which is already embedded in the FIT file. The pipeline globs *.fit only.

NDJSON parser (ndjson_parser.py)

NDJSON (newline-delimited JSON) means each line is an independent JSON object. Each line has the shape {"timestamp": "...", "entryData": {...}}. Four parse functions handle the four data types:

FunctionSource folderKey transforms
parse_sleep_line() sleep/ HR fraction → bpm (multiply by hr_rest); extracts _bedtime_end for stage linking; deduplicates repeated sessions
parse_sleep_stage_line() sleep_stages/ Lowercases stage enum (DEEP → deep)
parse_activity_line() activity/ Passes through step count and energy; no transformation needed
parse_recovery_line() recovery/ Passes through balance (0–1) and stress state

HR normalization — a critical detail

The Suunto sleep export stores heart rate as a fraction of resting HR, not as bpm. A value of 0.9 means 90% of resting HR. The pipeline multiplies by hr_rest (loaded from user/user.json) to convert to bpm before storing:

'hr_avg': round(hr_avg_raw * hr_rest) if hr_avg_raw is not None else None
Note: The original CLAUDE.md said to multiply by hrMax, which would produce values of 178–221 bpm during sleep — physiologically impossible. The code correctly uses hrRest. The documentation has been superseded by the implementation.

Sleep stage linking

All sleep stage rows are loaded into memory first, sorted by timestamp. For each sleep session, the pipeline finds all stage rows whose recorded_at falls within [bedtime_start, bedtime_end) and inserts them with the session's database id as the foreign key. This avoids a complex JOIN during ingestion.

Alembic Migrations alembic/

Alembic is the migration tool for SQLAlchemy. Every schema change — adding a column, creating a table, adding an index — must go through a migration file. The database is never edited manually.

Why this matters

Without migrations, the DB schema and the ORM models can silently diverge. A new column added to a model but not to the DB will cause crashes at runtime with no clear error. Alembic keeps a version table in the DB (alembic_version) and tracks exactly which migrations have been applied.

Agent Loop app/agent/agent.py

The agent is a loop that drives a conversation with the LLM until it produces a final text answer. It is the core intelligence of the system.

ask(question)
  │
  ├─ Build messages: [system prompt, user question]
  │
  ▼
┌─────────────────────────────────────┐
│ ollama.chat(model, messages, tools) │ ← LLM call
└─────────────────────────────────────┘
  │
  ├─ if no tool_calls → return msg.content ← done
  │
  ├─ if tool_calls present:
  │    ├─ dispatch each call to TOOL_DISPATCH[name](**args)
  │    ├─ append {"role": "tool", "content": json.dumps(result)}
  │    └─ loop back ↑
  │
  └─ (repeat until no tool calls)

The system prompt

The system prompt is injected at the start of every conversation. It establishes the model's role, tells it today's date (so it can compute relative date ranges like "last week"), and instructs it to always fetch data before answering:

"""You are Atlas, a personal physiological intelligence assistant for an elite multi-sport athlete.
You have access to the athlete's workout, sleep, activity, and recovery data.
Today's date is {today}.
When answering questions, call the relevant tools to fetch data before responding.
Be concise and specific. Use numbers from the data, not generic advice."""

Why a loop instead of a single call?

A single LLM call with tool definitions returns either a text answer or a set of tool calls — not both. When the model decides it needs data, it returns tool calls. The loop executes those calls, appends the results to the message history, and calls the model again. The model now has the data in context and can write a grounded answer. Without the loop, tool calls would just be returned to the caller and never executed.

Tool call dispatch

The model returns a tool name (a string) and arguments (a dict). The agent looks up the name in TOOL_DISPATCH, a plain Python dict mapping names to callables. If the name is unknown or the function raises, the error is serialized to JSON and appended as a tool result — the loop continues, allowing the model to recover.

Tools app/agent/tools.py

Tools are the agent's hands. They are typed Python functions that query the database and return plain dicts. Each tool also has a JSON schema definition that is sent to the LLM so it knows what tools exist and what arguments to pass.

The 5 MVP tools

ToolArgumentsReturnsAggregates?
get_workout_summary start_date, end_date, sport_type? List of workouts with duration, distance, HR zones No — one row per session
get_sleep_summary start_date, end_date List of sessions with deep/REM/light durations, HRV, SpO2, quality No — one row per session
get_activity_summary start_date, end_date Daily totals: steps + energy Yes — sums 30-min slots by day
get_recovery_summary start_date, end_date Recovery balance (0–1) and stress state per entry No — one row per reading
get_user_profile none Static profile: hrMax, hrRest, weight, height, DOB, gender N/A — reads a JSON file

How the LLM knows about the tools

Each tool has a JSON Schema definition in TOOL_SCHEMAS. This list is passed directly to ollama.chat(tools=TOOL_SCHEMAS). The model uses these schemas to decide which tool to call and to generate correctly-shaped argument dicts. The schema defines required vs optional parameters, their types, and a plain-English description of what the tool does.

# Example: the schema for get_workout_summary
{
  "type": "function",
  "function": {
    "name": "get_workout_summary",
    "description": "Get workouts over a date range, optionally filtered by sport type.",
    "parameters": {
      "type": "object",
      "properties": {
        "start_date": {"type": "string", "description": "Start date in ISO format, e.g. 2026-04-01"},
        "end_date":   {"type": "string", "description": "End date in ISO format, e.g. 2026-04-10"},
        "sport_type": {"type": "string", "description": "Optional sport type slug"}
      },
      "required": ["start_date", "end_date"]
    }
  }
}

Why tool-calling instead of RAG?

Retrieval-Augmented Generation (RAG) embeds text chunks and retrieves semantically similar ones. That works well for unstructured text (documents, notes). For numerical time-series data, it is unreliable — semantic similarity does not help you find "the three nights with lowest HRV" or "total distance cycled in March". Tool calling gives the model a precise, queryable interface to structured data.

Ollama & the Model qwen2.5:32b

Ollama is a local model server. It exposes an OpenAI-compatible HTTP API and handles model loading, GPU memory management, and inference. The Python ollama library is a thin client over this API.

Why qwen2.5:32b?

Tool-calling quality is the primary selection criterion. The model must reliably:

  • Infer the right date range from a relative question ("last two weeks")
  • Call the right tools in the right order
  • Synthesize multiple tool results into a coherent, quantitative answer

Smaller models (7B–14B) tend to hallucinate tool arguments or fail to call tools at all for complex queries. qwen2.5:32b is a strong balance of capability and local runtime feasibility at 19 GB.

The model's role vs the tools' role

The model does no computation. It does not calculate averages, filter arrays, or sort data. All of that happens in Python tool functions that run against the DB. The model's job is to understand the question, decide which tools to call, and narrate the results in natural language. This separation is important: if an answer is wrong, you can check whether it was a DB query bug (deterministic, testable) or a model reasoning error (harder to debug).

FastAPI Endpoint app/api/ask.py

The API is minimal by design. It is a thin HTTP wrapper around the agent — one route, two Pydantic models, no auth.

@router.post("/ask", response_model=AskResponse)
def ask(request: AskRequest) -> AskResponse:
    answer = agent.ask(request.question)
    return AskResponse(answer=answer)

Request / Response

DirectionShape
Request{"question": "string"}
Response{"answer": "string"}
 
# Example call:
curl -s -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "How was my sleep this week?"}'

Why FastAPI?

FastAPI generates automatic OpenAPI documentation at /docs (Swagger UI) and /redoc. Pydantic models on request/response mean the endpoint validates inputs automatically — a missing or wrong-typed field returns a 422 before the agent is ever called. For a single-user local tool this is lightweight overhead that pays back in debuggability.

Known limitation: The /ask handler and agent.ask() are currently synchronous. The LLM call can take 10–30 seconds for complex questions. Uvicorn serves this in a thread pool (via Starlette's default sync handling), so it won't block the event loop — but a proper async implementation with async def ask() and an async Ollama client would be cleaner.

Entry point: app/main.py

app = FastAPI(title="Atlas")
app.include_router(ask.router)

FastAPI uses routers to register groups of routes. ask.router is an APIRouter that holds the /ask endpoint. Adding new feature areas (e.g. a /ingest endpoint) would follow the same pattern: create a new router file and include it here.

Tests tests/

The test suite covers the three most failure-prone areas of the system.

FileWhat it testsHow
test_ndjson_parser.py All 4 NDJSON parse functions + iter_ndjson Pure unit tests — no DB. Constructs sample dicts, asserts output fields and edge cases.
test_fit_parser.py FIT filename parsing, session extraction, missing fields Mocks fitparse.FitFile — no real FIT files needed.
test_tools.py All 5 agent tools — date filtering, aggregation, field mapping SQLite in-memory DB via conftest.py. SessionLocal is patched to use the test session.
test_agent.py Agent loop termination, tool dispatch, error handling Mocks ollama.chat and TOOL_DISPATCH. No LLM needed.

Why these tests matter for this specific project

Atlas has no human in the loop between ingestion and the agent's answer. A bug in HR normalization, a wrong date filter, or a silent parser crash doesn't produce an error — it produces a plausible-sounding wrong number. The tests are the only automated check that the data reaching the model is correct.

How the Parts Connect

Here is the complete path of a single request, tracing every file involved:

StepFileWhat happens
1app/main.pyFastAPI app starts, registers ask.router
2app/api/ask.pyPOST /ask receives {"question": "..."}, validates via Pydantic
3app/agent/agent.pyBuilds message list with system prompt + user question, calls ollama.chat()
4app/agent/tools.pyOllama returns tool calls; agent dispatches via TOOL_DISPATCH
5app/core/database.pyTool opens a SessionLocal() context
6app/models/*.pyORM query runs against Postgres; rows returned as model instances
7app/agent/tools.pyModel instances serialized to plain dicts, returned to agent
8app/agent/agent.pyResult JSON appended to messages; loop calls ollama.chat() again
9app/agent/agent.pyModel returns no more tool calls; msg.content is the final answer
10app/api/ask.pyAnswer wrapped in AskResponse, returned as JSON

Configuration flows through app/core/config.py

Pydantic Settings reads .env at startup and exposes a single settings singleton. Three files import it: database.py (for the DB URL), agent.py (for the Ollama model name), and tools.py (for the export directory path used by get_user_profile). Changing the model or database requires only an .env edit — no code changes.

The ingestion pipeline is a one-way door

Ingestion writes to the DB. Nothing else does. The agent tools, the API, the tests — all read only. This means the DB state at any point is purely a function of the export data and the pipeline code. To rebuild it from scratch: drop the tables, run migrations, run the pipeline.
